1. Crea un primer archivo de `workflow` de GitHub Actions.
   (ej: `.github/workflows/fork_pr_benchmarks_track.yml`)
2. Nombra este workflow como segundo workflow.
   (ej: `name: Track Benchmarks with Bencher`)
3. Encadena los dos workflows con
   [el evento `workflow_run`][github actions workflow_run].
   (ej: `on: workflow_run: ...`)
4. Crea un `job` de GitHub Actions.
   (ej: `jobs: track_fork_pr_branch`)
5. Ejecuta este job solo si la conclusión del workflow anterior fue un éxito usando
   [el evento `workflow_run` de GitHub Actions][github actions workflow_run].
   (ej: `if: github.event.workflow_run.conclusion == 'success'`)
6. Establece [los permisos para el `GITHUB_TOKEN`][github token permissions]
   como `write` para `pull-requests`.
   Dependiendo de la configuración de tu GitHub, esto puede no ser necesario.
   Pero para todas las organizaciones y repositorios personales
   [creados después del 02 de febrero de 2023][github token read only],
   este es el comportamiento predeterminado.
   Consulta [la documentación de GitHub][github token permissions security]
   para una visión completa.
   (ej: `permissions: pull-requests: write`)
7. Establece el tipo de máquina en la que el job se ejecutará.
   Consulta la [documentación de `runs-on` de GitHub Actions][github actions runs-on]
   para una visión completa.
   (ej: `runs-on: ubuntu-latest`)
8. Establece los resultados de las pruebas y los nombres de los archivos de objetos de eventos `pull_request` como variables de entorno.
   (ej: `env: ...`)
9. Descarga los resultados de pruebas almacenados en caché y el evento `pull_request`
   usando [la acción GitHub `action-download-artifact`][action download artifact].
   (ej: `uses: dawidd6/action-download-artifact@v6`)
10. Exporta los datos necesarios desde el evento `pull_request` como [variables de entorno intermedias][github intermediate environment variable].
    (ej: `core.exportVariable(...)`)
11. Instala el CLI de Bencher usando [la acción de GitHub][bencher cli github action].
   (ej: `uses: bencherdev/bencher@main`)
12. Usa el subcomando CLI <code><a href="/es/docs/explanation/bencher-run/">bencher run</a></code>
    para llevar un seguimiento de los benchmarks de la rama de pull request del fork.
    Consulta [el subcomando CLI `bencher run`][bencher run] para una visión completa.
    (ej: `bencher run`)
13. Establece la opción `--project` al slug del Proyecto.
    Consulta [los documentos de `--project`][project option] para más detalles.
    (ej: `--project project-abc4567-wxyz123456789`)
14. Establece la opción `--token` al secreto del **Repositorio** `BENCHER_API_TOKEN`.
    Consulta [los documentos de `--token`][token option] para más detalles.
    (ej: `--token '${{ secrets.BENCHER_API_TOKEN }}'`)
15. Establece la opción `--branch` al nombre de la rama PR del fork
    usando [una variable de entorno intermedia][github intermediate environment variable].
    Consulta [los documentos de `--branch`][branch option] para una visión completa.
    (ej: `--branch "$PR_HEAD"`)
16. Establece la opción `--hash` al hash `git` de la rama PR del fork
    usando [una variable de entorno intermedia][github intermediate environment variable].
    Consulta [los documentos de `--hash`][hash option] para una visión completa.
    (ej: `--hash "$PR_HEAD_SHA"`)
17. Establece el Punto de Inicio para la Ramas PR del fork:
    1. Establece la opción `--start-point` al punto de inicio de la Ramas PR del fork
    usando [una variable de entorno intermedia][github intermediate environment variable].
    Consulta [los documentos de `--start-point`][start point] para una visión completa.
    (ej: `--start-point "$PR_BASE"`)
    2. Establece la opción `--start-point-hash` al hash `git` del punto de inicio de la Ramas PR del fork
    usando [una variable de entorno intermedia][github intermediate environment variable].
    Consulta [los documentos de `--start-point-hash`][start point hash] para una visión completa.
    (ej: `--start-point-hash "$PR_BASE_SHA"`)
    3. Establece el flag `--start-point-clone-thresholds` para clonar las Umbrales desde el punto de inicio.
    Consulta [los documentos de `--start-point-clone-thresholds`][start point clone thresholds] para una visión completa.
    (ej: `--start-point-clone-thresholds`)
    4. Establece el flag `--start-point-reset` para siempre restablecer la Ramas PR del fork al punto de inicio.
    Esto evitará la deriva de datos de benchmark.
    Consulta [los documentos de `--start-point-reset`][start point reset] para una visión completa.
    (ej: `--start-point-reset`)
18. Establece la opción `--testbed` al nombre del Entorno de Pruebas.
    Es probable que esto coincida con la máquina seleccionada en `runs-on`.
    Consulta [los documentos de `--tested`][testbed option] para más detalles.
    (ej: `--testbed ubuntu-latest`)
19. Establece el flag `--err` para que el comando falle si se genera una Alerta.
   Consulta [los documentos de `--err`][alert err] para una visión completa.
   (ej: `--err`)
20. Establece la opción `--adapter` al [Formato de Métrica Bencher JSON (`json`)][bmf] que es generado por <code><a href="/es/docs/reference/bencher-metric-format/#bencher-mock">bencher mock</a></code>.
   Consulta [los adaptadores de harness de benchmark][adapter json] para una visión completa.
   (ej: `--adapter json`)
21. Establece la opción `--github-actions` al token de autenticación de la API de GitHub para publicar resultados como un comentario en la Pull Request usando
    [la variable de entorno `GITHUB_TOKEN` de GitHub Actions][github token].
    Consulta [los documentos de `--github-actions`][github actions option] para más detalles.
    (ej: `--github-actions '${{ secrets.GITHUB_TOKEN }}'`)
22. Establece la opción `--ci-number` al número de la pull request
    usando [una variable de entorno intermedia][github intermediate environment variable].
    Consulta [los documentos de `--ci-number`][ci number option] para más detalles.
    (ej: `--ci-number "$PR_NUMBER"`)
23. Establece la opción `--file` a la ruta del archivo de resultados de benchmark.
    Consulta [comando de benchmark][command argument] para una visión completa.
    (ej: `--file "$BENCHMARK_RESULTS"`)
