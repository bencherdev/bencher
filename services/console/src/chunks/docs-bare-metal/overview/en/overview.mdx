Most benchmark tracking tools only let you run benchmarks from a single environment.
Either you run your benchmarks locally or you run them in CI.
Even if they support both local and CI execution,
you're unable to compare results between the two.

Bencher Bare Metal allows you to have the best of both worlds.
You can run benchmarks from your local machine and have them executed on dedicated hardware,
and you can also run benchmarks from CI and have them executed on the _same_ dedicated hardware.
This allows you to have a consistent benchmark execution environment regardless of where you run your benchmarks.

Bencher Bare Metal runs on dedicated hardware to eliminate variability and provide more accurate and reliable benchmark results.

## Runner

A Runner is a remote benchmark execution orchestrator that runs on dedicated, bare metal hardware.
The Runner polls the Bencher API for pending [Jobs][job] based on its associated [Specs][spec].
Once a Job is claimed, the Runner pulls the specified [Image][image] for the Job,
unpacks the Image, executes the benchmark command, and sends the results back to the Bencher API.
Only one Job is executed at a time to ensure that the benchmarks are not affected by other workloads.

## Spec

A Spec is a hardware specification that describes the resources available on a Runner.

1. Operating System (OS)
   1. Linux
   2. macOS
   3. Windows
2. CPU Architecture
   1. x86_64
   2. ARM64
3. [Sandbox][sandbox]
   1. Firecracker microVM
   2. No Sandbox
4. CPU Count
5. Memory Size (RAM)
6. Disk Size
7. Network Access

[sandbox]: #sandbox

## Sandbox

A Sandbox is an isolation mechanism that allows for safe multi-tenancy on a Runner.
Bencher Bare Metal uses [Firecracker][firecracker] microVMs as the default Sandbox implementation.
Running without a Sandbox on Bencher Cloud requires a Dedicated or Custom Runner (contact <a href="mailto:plus@bencher.dev">Bencher Plus support</a> for more information) or Bencher Self-Hosted.
When a Runner is configured to use a Sandbox, each Job is executed within that Sandbox.
See [the `runner` documentation][runner] for a full overview.

[runner]: /docs/bare-metal/runner

## Job

A Job is a benchmark execution unit.
When you run <code><a href="/docs/explanation/bencher-run/">bencher run</a></code> with the `--image` and `--spec` options,
a Job is created and placed in the queue.
A Runner with a matching Spec claims the Job, executes the benchmark container, and reports the results.

## Image

An Image is an [OCI container image][oci image] that contains your benchmark code and its dependencies.
When a Runner executes a Job, it pulls the Image and runs it in an isolated container environment.
See the [Image page][image page] for how to create an Image for your own project.

## Bare Metal vs Local Execution

With local execution, <code><a href="/docs/explanation/bencher-run/">bencher run</a></code> executes benchmarks directly on the current machine
and reports the results to the Bencher API.
This is the default behavior and works well for CI environments and local development.

With Bare Metal execution, <code><a href="/docs/explanation/bencher-run/">bencher run</a></code> packages the benchmark command,
submits it as a Job to the Bencher API, and waits for a remote Runner to execute it on dedicated hardware.
The results are then reported back through the same pipeline.
To use Bare Metal execution, add the `--image` and `--spec` options to your `bencher run` command.
