## What is a Spec?

A Spec is a hardware specification that describes the resources available for benchmark execution on a Bare Metal Runner.
Each Spec defines:

- **OS**: The operating system (e.g. `linux`, `macos`, `windows`)
- **Architecture**: The CPU architecture (e.g. `x86_64`, `aarch64`)
- **CPU**: The number of CPUs
- **Memory**: Memory size in bytes
- **Disk**: Disk size in bytes
- **Network**: Whether the environment has network access
- **Sandbox**: The sandbox type (e.g. `firecracker`), or none for unsandboxed execution

The OS and sandbox fields are immutable after creation.
All other fields (name, slug, fallback) can be updated.

Specs are server-scoped, so the same Spec can be shared across multiple Runners and Testbeds.

## Available Cloud Specs

Bencher Cloud provides pre-configured Specs for common hardware configurations.
For example, `intel-v1` is a Linux x86_64 Spec with Firecracker sandboxing.
Use `bencher spec list` to see all available Specs.

## Choosing a Spec

Select a Spec that matches the resource requirements of your benchmarks.
Consider the CPU count, memory, and disk space your benchmarks need.
If your benchmarks require network access (e.g. to download dependencies), choose a Spec with `network: true`.

When submitting a benchmark run with `bencher run --image ... --spec SPEC`,
the Spec determines the hardware resources for the job.
If no `--spec` is provided, the server's fallback Spec is used.

## Spec and Testbed Naming

When `--image` is used to create a Bare Metal Job, the Spec determines the default Testbed name.
The following table shows how the Testbed name is resolved:

| [`--testbed`][testbed] provided | [`--image`][image] provided | Testbed name source |
|---|---|---|
| No | No | Context OS |
| Yes | No | User's explicit value |
| No | Yes | Resolved Spec's name (Spec assigned to Testbed) |
| Yes | Yes | User's explicit value (Spec assigned to Testbed) |

[testbed]: /docs/explanation/bencher-run/#--testbed-testbed
[image]: /docs/explanation/bencher-run/#--image-image

## Sandboxed vs Non-Sandboxed Execution

Specs with a `sandbox` value of `firecracker` run benchmarks inside a Firecracker microVM.
This provides strong isolation: each job gets its own lightweight VM with dedicated CPU, memory, and disk.
Firecracker sandboxing requires a Linux host with KVM support.

Specs without a `sandbox` value run benchmarks directly on the host.
The Runner extracts the OCI image layers and executes the benchmark command in a `chroot` environment.
Non-sandboxed execution is useful when Firecracker is not available (e.g. macOS or Windows hosts),
but provides weaker isolation.

## Managing Specs via API

Specs can be managed through the [Specs REST API][specs api].
All Spec endpoints require authentication with a Bencher API token.

[specs api]: /docs/api/server/#tag/specs

## Managing Specs via CLI

Use the `bencher` CLI to manage Specs:

- `bencher spec list` - List all Specs
- `bencher spec create` - Create a new Spec
- `bencher spec view SPEC` - View a Spec
- `bencher spec update SPEC` - Update a Spec
- `bencher spec delete SPEC` - Delete a Spec
