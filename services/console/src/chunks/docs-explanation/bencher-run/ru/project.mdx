### `--project <PROJECT>`

<br />

Необязательно: либо опция `--project`, либо переменная окружения `BENCHER_PROJECT`
может быть установлена в значение slug или UUID Проекта.
[Ключ API проекта](#--key-key) нельзя использовать для создания нового Проекта или присвоения существующего Проекта, поэтому требуется, чтобы `--project`/`BENCHER_PROJECT` была установлена в существующий Проект.
Если указан slug и Проект ещё не существует, он будет создан автоматически, когда используется ключ API пользователя ([опция `--key`](#--key-key)), [устаревшая опция `--token`](#--token-token) или учетные данные не предоставлены.
Если же указано UUID, Проект должен уже существовать.
Если указаны оба варианта, опция `--project` имеет приоритет над переменной окружения `BENCHER_PROJECT`.
Если ни один не указан, slug Проекта будет сгенерирован автоматически на основе:

1. имени родительской директории репозитория `git`, если он доступен.
2. 7-значного шестнадцатеричного короткого хеша начального коммита репозитория `git`, если он доступен.
3. 13-значного буквенно-цифрового отпечатка локальной машины для поддерживаемых операционных систем.

Например, сгенерированный slug Проекта может выглядеть так: `project-abc4567-wxyz123456789`

Когда для вас создаётся новый Проект, независимо от того, указан ли slug вручную или сгенерирован,
Проект должен принадлежать Организации.
Если пользователь аутентифицирован с помощью ключа API пользователя ([опция `--key`](#--key-key)) или [устаревшей опции `--token`](#--token-token), Проект будет добавлен в личную Организацию этого пользователя.
Если пользователь не аутентифицирован, Проект будет создан в новой Организации `unclaimed`.
Эту Организацию затем можно `claimed` с публичной Perf Page проекта
или, используя slug Проекта при аутентификации в последующем вызове `bencher run`.

> 🐰 ВАЖНО: Если переменная окружения `CI` установлена в `true`,
> то вам нужно либо указать проект,
> либо установить [флаг `--ci-on-the-fly`][ci on the fly].
