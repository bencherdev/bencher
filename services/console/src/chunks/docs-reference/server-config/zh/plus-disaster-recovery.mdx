### `plus.disaster_recovery`

此部分指定了灾难恢复配置。
Bencher支持连续复制所有数据库更改。
欲了解按需运行或计划备份，请参阅配置中的`database.data_store`部分。

有三种复制`scheme`：
- `file`: 复制到本地文件路径
    - `path`: 复制路径
- `sftp`: 通过SFTP复制
    - `host`: 目标系统主机名
    - `port`: 目标系统端口号
    - `user`: 目标系统的用户名
    - `password`: （可选）目标系统的密码
    - `path`: （可选）目标系统中的路径
    - `key_path`: （可选）SSH密钥路径
- `s3`: 复制到任何S3兼容的Blob存储
    - `bucket`: 存储桶名称
    - `path`: （可选）存储桶中的路径
    - `endpoint`: （AWS：可选 | 非AWS：必需）复制端点
    - `region`: （可选）存储桶区域
    - `access_key_id`: S3访问密钥
    - `secret_access_key`: S3密钥

以上三种复制`scheme`均有以下附加选项：

- `snapshot.interval`: （可选）指定创建新快照的频率。这用于减少恢复时间，因为较新的快照将有较少的WAL帧需要应用。保留仍然适用于这些快照。如果未设置快照间隔，则每次执行保留时都会创建新快照。默认情况下，每24小时进行一次保留。
- `snapshot.retention`: （可选）快照和WAL文件保留的时间。保留期过后，会创建新的快照并删除旧的。在最旧快照之前存在的WAL文件也将被删除。默认值为`24h`。
- `validation.interval`: （可选）如果指定，Bencher将自动恢复并验证副本上的数据与本地副本是否匹配。默认情况下禁用。启用此选项可能会显著增加运行Bencher的成本，因为大多数云服务会为下载收费。
- `sync_interval`: （可选）将帧推送到副本的频率。默认值为`1s`。增加频率可能显著增加云存储成本。
- `checkpoint.interval`: （可选）Litestream执行非阻塞PASSIVE检查点的频率，不受页面数量影响。如果有活跃的读取器或写入器，Litestream将跳过检查点。默认值为`1m`。
- `checkpoint.min_page_count`: （可选）触发非阻塞PASSIVE检查点前的最小WAL页面数（每页约4KB）。如果有活跃的读取器或写入器，Litestream将跳过检查点。默认值为`1000`（约4MB）。
- `checkpoint.truncate_page_n`: （可选）阻塞TRUNCATE检查点的页面数阈值。当达到此阈值时，Litestream会执行阻塞检查点，等待所有读取器和写入器完成，这可能在负载下导致`database is locked`错误。**默认值为`0`（禁用）。** 仅在需要WAL文件大小上限并了解阻塞影响的情况下才设置为非零值。

| :------------------------: | :-------: | :----: | :------: | :-----------------------------------------------------: |
|          replica           | \{ ... \} |  ---   |    是    |                     指定副本对象。                      |

|     snapshot.interval      |   "1h"    |  ---   |    否    | 创建新快照的频率。减少恢复时间。 |
|     snapshot.retention     |  "24h"    |  24h   |    否    | 快照和WAL文件在被替换前保留的时间。 |
|    validation.interval     |   "6h"    |  ---   |    否    | 恢复并验证副本数据与本地副本一致性的频率。默认禁用。 |
|    checkpoint.interval     |   "1m"    |   1m   |    否    | 非阻塞PASSIVE检查点的执行频率。如果有活跃的读取器/写入器则跳过。 |
