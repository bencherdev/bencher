All Bare Metal features are Bencher Plus features.

## When to Self-Host

Self-hosted Runners are appropriate when you need:

- **Data sovereignty**: Benchmark code and results never leave your infrastructure
- **Custom hardware**: Specific CPU models, GPUs, or other hardware not available on Bencher Cloud
- **Air-gapped environments**: Networks with no external internet access
- **Full control**: Custom kernel configurations, security policies, or compliance requirements

## Prerequisites

- A **Linux host** (required for the Runner binary)
- **KVM support** (required for Firecracker sandbox mode; check with `ls /dev/kvm`)
- A running **Bencher Self-Hosted** API server with Bencher Plus enabled

If you do not need Firecracker sandboxing, the KVM requirement can be relaxed,
but you must explicitly opt in with the `--danger-allow-no-sandbox` flag.

## Installing the Runner Binary

Download the `runner` binary from the [Bencher releases](https://github.com/bencherdev/bencher/releases)
or build from source:

```
cargo build --release -p bencher_runner_cli --features plus
```

The binary is located at `target/release/runner`.

## Server Configuration

The Bencher API server requires two configuration sections for Bare Metal Runner support.

### Runner Configuration

The `plus.runners` section controls job lifecycle timeouts:

```toml
[plus.runners]
# Seconds without a heartbeat before marking a job as failed (default: 90)
heartbeat_timeout = 90
# Extra seconds beyond a job's configured timeout before the server cancels it (default: 60)
job_timeout_grace_period = 60
```

See the [Server Config reference][server config runners] for all options.

[server config runners]: /docs/reference/server-config/#plusrunners

### Registry Configuration

The `plus.registry` section configures the OCI container registry
that Runners use to pull benchmark images:

```toml
[plus.registry]
# External URL of the API server for OCI registry access
url = "https://bencher.example.com"
```

The registry supports local filesystem storage (default) or AWS S3 for blob storage.
See the [Server Config reference][server config] for details.

[server config]: /docs/reference/server-config/

## Creating a Runner

Register a Runner with the API server:

```
bencher runner create --name my-runner --host https://bencher.example.com
```

The response includes a runner token. This token is only displayed once.
Store it securely.

## Token Management

If you need to rotate a runner token:

```
bencher runner token my-runner --host https://bencher.example.com
```

This invalidates the previous token and returns a new one.
Runner tokens are SHA-256 hashed at rest in the database --
the plaintext token is never stored on the server.

## Associating Specs

Create Specs that describe your hardware, then associate them with the Runner.

Create a Spec:

```
bencher spec create \
  --name my-spec \
  --os linux \
  --architecture x86_64 \
  --sandbox firecracker \
  --cpu 4 \
  --memory 8589934592 \
  --disk 10737418240 \
  --host https://bencher.example.com
```

Associate it with the Runner:

```
bencher runner spec add my-runner --spec my-spec --host https://bencher.example.com
```

## Starting the Runner Daemon

Start the Runner with:

```
runner up --runner my-runner --token <TOKEN> --host https://bencher.example.com
```

For non-sandboxed execution (no Firecracker microVM):

```
runner up --runner my-runner --token <TOKEN> --host https://bencher.example.com --danger-allow-no-sandbox
```

### Runner Flags

| Flag | Environment Variable | Description |
|---|---|---|
| `--runner` | `BENCHER_RUNNER` | Runner UUID or slug |
| `--token` | `BENCHER_RUNNER_TOKEN` | Runner authentication token |
| `--host` | `BENCHER_HOST` | Bencher API server URL |
| `--poll-timeout` | | Long-poll timeout in seconds (1-900, default 55) |
| `--danger-allow-no-sandbox` | `BENCHER_DANGER_ALLOW_NO_SANDBOX` | Allow non-sandboxed execution |
| `--grace-period` | | Grace period in seconds after exit code |
| `--max-output-size` | | Max stdout/stderr size in bytes (default 25 MiB) |
| `--max-file-count` | | Max output files to collect (default 255) |

## Monitoring

The Runner sends periodic heartbeat messages to the API server over the WebSocket connection.
The server tracks the last heartbeat timestamp for each Runner.

If a heartbeat is not received within the configured `heartbeat_timeout`,
the server marks the active Job as Failed and closes the connection.

Monitor your Runner's status with:

```
bencher runner view my-runner --host https://bencher.example.com
```

The `last_heartbeat` field in the response indicates the most recent heartbeat time.

## Security

- **Runner tokens** are SHA-256 hashed at rest. The plaintext is only returned on creation or rotation.
- **OCI tokens** issued to Runners are short-lived JWTs scoped to a single project. They are generated at job claim time and expire shortly after the job timeout.
- Firecracker microVMs provide hardware-level isolation between benchmark executions. Each job runs in its own VM with dedicated resources.
